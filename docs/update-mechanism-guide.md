# Инструкция: механизм самообновления PWA (перенос из «Таблицы умножения»)

Задача: внедрить в PWA-приложение (викторина по Minecraft) проверенную систему обновлений:
пользователь с «замороженным» в памяти телефона приложением получает новую версию сам —
тихо или через плашку, — и обновление работает даже на слабой сети.

## Предпосылки (проверь, что совпадает с твоим приложением)

- Статический сайт без сборки (например, один `index.html` + `sw.js`), хостинг отдаёт
  заголовки `ETag` и/или `Last-Modified` (GitHub Pages — отдаёт).
- Есть Service Worker, кэширующий оболочку приложения (app shell) в Cache Storage,
  HTML отдаётся по стратегии network-first с таймаутом и фолбэком на кэш.
- UI на любом фреймворке; ниже примеры в духе React, легко адаптировать.

Если SW нет вообще — сначала сделай базовый офлайн-кэш, механизм опирается на него.

## Идея в одном абзаце

Раз в 15 минут и при возврате приложения из фона сравниваем `ETag` свежего `index.html`
на сервере (лёгкий HEAD-запрос мимо кэшей) с `ETag` копии, лежащей в кэше SW. Разошлись —
есть новая версия. Дальше по контексту: на «спокойном» экране тихо обновляемся сами;
на «занятом» показываем плашку с кнопкой; во время игры/ввода не мешаем вообще. Само
обновление: под перекрывающим оверлеем «Обновляю…» просим SW скачать свежий HTML В КЭШ
и перезагружаемся ТОЛЬКО после подтверждения — поэтому работает и на еле живой сети.

## Компонент 1: команда `refresh-shell` в sw.js

```js
self.addEventListener('message', function(e) {
  if (e.data && e.data.type === 'refresh-shell') {
    var port = e.ports && e.ports[0];
    fetch('./index.html', { cache: 'reload' }).then(function(resp) {
      if (!resp || resp.status !== 200) throw new Error('bad status');
      return caches.open(SHELL_CACHE).then(function(cache) {
        // КРИТИЧНО: навигация обычно обслуживается из ключа '.', а прямые
        // запросы — из './index.html'. Обнови ОБА ключа, которые есть в твоём
        // SHELL-списке, иначе на слабой сети перезагрузка возьмёт старый корень.
        return cache.put('./index.html', resp.clone()).then(function() {
          return cache.put('.', resp.clone());
        });
      });
    }).then(function() {
      if (port) port.postMessage({ ok: true });
    }).catch(function() {
      if (port) port.postMessage({ ok: false });
    });
  }
});
```

Почему нельзя «просто `location.reload()`»: при network-first с таймаутом (у нас 3с)
перезагрузка на медленной сети отдаст СТАРЫЙ кэш по таймауту — обновление молча не
произойдёт. Сначала кладём свежую версию в кэш, потом перезагружаемся: тогда даже
фолбэк на кэш отдаёт уже новую версию.

## Компонент 2: обнаружение новой версии (в коде страницы)

```js
// в чистой логике (тестируемо):
function normalizeEtag(v) { return v ? String(v).replace(/^W\//, "") : null; }

// в приложении:
const checkUpdate = async () => {
  if (updatingRef.current || !navigator.onLine || document.hidden) return;
  try {
    const head = await fetch("./index.html", { method: "HEAD", cache: "no-store" });
    if (!head.ok) return;
    const net = normalizeEtag(head.headers.get("etag")) || head.headers.get("last-modified");
    if (!net) return;
    const cached = await caches.match("./index.html");
    if (!cached) return; // офлайн-кэш ещё не построен — обновлять нечего
    const cur = normalizeEtag(cached.headers.get("etag")) || cached.headers.get("last-modified");
    if (cur && net !== cur) setUpdAvail(true);
  } catch(e) {} // сеть моргнула — проверим в следующий раз
};

// триггеры:
useEffect(() => {
  const t0 = setTimeout(checkUpdate, 8000);            // после старта, не мешая загрузке
  const iv = setInterval(checkUpdate, 15 * 60 * 1000); // раз в 15 минут
  const onVis = () => { if (!document.hidden) checkUpdate(); }; // возврат из фона
  document.addEventListener("visibilitychange", onVis);
  return () => { clearTimeout(t0); clearInterval(iv);
    document.removeEventListener("visibilitychange", onVis); };
}, []);
```

Нюансы:
- HEAD не перехватывается типичным SW (`if (e.request.method !== 'GET') return;`) —
  проверь, что у тебя так же, иначе сравнение пойдёт с кэшем.
- Ответ кэша хранит заголовки оригинального ответа — `cached.headers.get("etag")` работает.
- Никаких «version.json» вести не нужно: сервер отдаёт ETag бесплатно.

## Компонент 3: контекст экранов — когда можно обновляться

Экраны публикуют один из трёх контекстов (простейшая шина на window):

```js
window._updCtx = "banner"; // дефолт для экранов, которые ничего не сообщают
window._setUpdCtx = v => {
  if (window._updCtx !== v) { window._updCtx = v; window.dispatchEvent(new Event("updctx")); }
};
```

- `"auto"` — главное меню, пользователь ничего не делает: можно тихо перезагрузиться.
- `"banner"` — результаты, настройки, форма входа: показать плашку «Обновить».
- `"hide"` — идёт викторина/ввод ответа: не показывать НИЧЕГО, не мешать.

Каждый экран ставит свой контекст в эффекте (пример: меню → auto, вопрос → hide),
и сбрасывает на "banner" при размонтировании. Для викторины: экран вопроса = hide,
экран между раундами/итогов = banner, главное меню = auto.

## Компонент 4: решение — чистая функция с защитой от циклов

```js
function updateAction(o) {
  if (!o || !o.available || o.updating) return "none";
  if (o.ctx === "hide") return "none";
  if (o.ctx !== "auto") return "banner";
  if (o.sinceFailMs != null && o.sinceFailMs < 60000) return "banner";   // недавний провал сети
  if (o.sinceAutoMs != null && o.sinceAutoMs < 120000) return "banner";  // недавнее автообновление
  return "auto";
}
```

Обе защиты обязательны: без них возможен цикл «перезагрузка → версия почему-то не
применилась → снова перезагрузка». После авто-перезагрузки пиши в localStorage метку
времени (`lastAutoUpd`), после провала скачивания — запоминай момент провала.
Автообновление запускай только если контекст «auto» продержался ~2.5 секунды
(пользователь реально бездействует, а не проскакивает меню по пути к игре).

## Компонент 5: само обновление + оверлей (блокирует действия)

```js
const startUpdate = async () => {
  if (updatingRef.current) return;
  updatingRef.current = true;
  setUpdating(true); // рисует полноэкранный оверлей поверх ВСЕГО (position:fixed; inset:0; максимальный z-index)
  try {
    const reg = await navigator.serviceWorker.getRegistration();
    if (reg) reg.update().catch(() => {}); // заодно браузер проверит sw.js
  } catch(e) {}
  const ok = await new Promise((resolve) => {
    const ctrl = navigator.serviceWorker && navigator.serviceWorker.controller;
    if (!ctrl) { resolve("no-sw"); return; } // SW нет — просто reload, хуже не будет
    const ch = new MessageChannel();
    const to = setTimeout(() => resolve(false), 30000);
    ch.port1.onmessage = (e) => { clearTimeout(to); resolve(!!(e.data && e.data.ok)); };
    ctrl.postMessage({ type: "refresh-shell" }, [ch.port2]);
  });
  if (ok === true || ok === "no-sw") {
    try { localStorage.setItem("lastAutoUpd", String(Date.now())); } catch(e) {}
    window.location.reload();
    return;
  }
  lastFailRef.current = Date.now(); // сеть подвела: убрать оверлей, плашка останется
  updatingRef.current = false;
  setUpdating(false);
};
```

Оверлей: крутящийся значок (CSS `@keyframes spin`), текст вроде «Обновляю приложение…
Пара секунд — и продолжим!». Он ДОЛЖЕН перекрывать весь экран и все модалки — на
слабой сети скачивание длится секунды, и пользователь не должен успеть ничего нажать.

Плашка (когда контекст "banner"): фиксированная полоска внизу экрана с текстом
«Появилась новая версия» и кнопкой «Обновить» → тот же `startUpdate()`. При
контексте "hide" плашка не рендерится вовсе.

## Компонент 6: видимый номер версии

Константа в коде: `const APP_VERSION = "ГГГГ-ММ-ДД.N";` (дата выкладки + номер выкладки
за день). Показывай мелким серым текстом на форме входа/в настройках/внизу главного
экрана — иначе невозможно ответить на вопрос «какая версия у меня стоит?» и проверить,
доехало ли обновление. ВАЖНО: заведи правило в CLAUDE.md проекта — обновлять
APP_VERSION при каждом пуше, меняющем код, и брать дату командой `date`
непосредственно перед коммитом (не по памяти — длинные сессии врут).

## Грабли, на которые мы уже наступили (не повторяй)

1. **Два ключа кэша.** Навигация и прямой запрос HTML — разные ключи в Cache Storage.
   Обновляй оба, иначе «обновился, а версия старая» на слабой сети.
2. **Перезагрузка до скачивания.** Никогда не делай reload по факту «обнаружил новую
   версию» — сначала скачай в кэш через SW, потом reload (см. Компонент 1).
3. **Циклы перезагрузок.** Обе защиты из Компонента 4 + метки времени.
4. **Обновление посреди игры/ввода.** Контекст "hide" для игровых экранов и полей
   ввода; для формы логина — только плашка (перезагрузка стёрла бы набранный пароль).
5. **localStorage может кидать исключения** (приватный режим Safari) — все обращения
   в try/catch.
6. **Первая доставка — вручную.** Сам механизм приедет к пользователям только когда
   они один раз холодно перезапустят приложение с интернетом. Все СЛЕДУЮЩИЕ версии —
   уже автоматически.

## Как проверить (обязательно, руками или e2e)

1. Открой приложение, дождись построения кэша. Выложи новую версию (поменяй APP_VERSION).
2. Сверни/разверни вкладку на главном меню → через ~2.5с приложение тихо перезагрузится,
   внизу — новый номер версии.
3. То же, но находясь на игровом экране → ничего не происходит; выйди в меню/итоги →
   плашка или тихое обновление.
4. Слабая сеть (Chrome DevTools → throttling Slow 3G): нажми «Обновить» на плашке →
   оверлей висит, пока качается, потом перезагрузка сразу на новую версию.
5. Обрыв сети во время скачивания → оверлей убирается, плашка остаётся, авто-повтор
   не раньше чем через минуту.
6. Убедись, что в тестах покрыта чистая логика `updateAction` (все контексты × защиты).
